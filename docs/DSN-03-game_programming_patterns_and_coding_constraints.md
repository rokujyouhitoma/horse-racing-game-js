# [DSN-03] ゲームプログラミングパターンとコーディング制約定義書 (Game Programming Patterns & Coding Constraints) - horse-racing-game-js

本ドキュメントは、「`horse-racing-game-js`」プロジェクトにおける実装の方針、およびそれに付随するコーディング上の制約を定義したものです。
本プロジェクトは、フレームワークに依存しない Vanilla JavaScript (ES5/ES6混在期) 環境で記述されており、メンテナンス性とパフォーマンス、堅牢性を最大化するために、名著『**Game Programming Patterns**』に記載されている各種デザインパターンをベースにした設計方針と厳格なコード制約を設けています。

---

## 1. 本書の目的

本ドキュメントの目的は以下の通りです。
1. **設計のブレ防止**: 開発者がコードを追加・修正する際、一貫したアーキテクチャ設計に則ってコーディングできるようにする。
2. **フレームワーク非依存での秩序維持**: ReactやVueといったフレームワークが提供する状態管理やレンダリング自動化の代わりに、デザインパターンを用いた秩序ある構造を維持する。
3. **バグと脆弱性の未然防止**: XSS脆弱性の完全排除、メモリリーク防止、コンパイル警告のゼロ化を強制する制約を明文化する。

---

## 2. Game Programming Patterns に基づく設計方針と制約

『[Game Programming Patterns](https://gameprogrammingpatterns.com/contents.html)』の分類に従い、本プロジェクトでの適用状況およびコーディング上の制約を定義します。

### 2.1 Design Patterns Revisited (再評価されたデザインパターン)

#### ① Command (コマンド)
* **適用箇所**:
  * プレイヤーのカードプレイ操作（`PlayCardCommand`）、および `CommandExecuter` / `SimpleCommandExecuter` によるコマンド管理。
* **コーディング制約**:
  * **直接的な状態変更の禁止**: レース中における馬の位置の前進・後退や、コイン・スコアなどの「ゲーム進行に直接関わる状態」を変更する操作は、すべて `ICommand` インターフェースを実装したコマンドクラスにカプセル化しなければならない。
  * **Undo可能性の保証**: すべてのコマンドクラスは `Execute()` に加え、状態を完全に元に戻す `Undo()` メソッドを必ず実装しなければならない。

#### ② Flyweight (フライウェイト)
* **適用箇所**:
  * `MasterData` および `StubLoader` から読み込まれる静的データ。`Model` クラス（`MasterMeta`）による属性データの共有。
* **コーディング制約**:
  * **静的データの複製禁止**: 各エンティティインスタンス（例: `HorseFigure`, `MonsterFigure` の個体）は、静的（読み取り専用）なマスタープロパティ（名前、カラーコード、初期スピードなど）のコピーを個別に保持してはならない。必ず `Model` インスタンスを参照する形に集約し、メモリ空間を節約すること。

#### ③ Observer (オブザーバー)
* **適用箇所**:
  * `Game.Publisher` による Pub/Sub 通信。
* **コーディング制約**:
  * **ビューとロジックの完全疎結合化**: 描画レイヤー（`RacetrackLayer`, `OddsTableLayer` など）がコントローラーやモデルを直接参照して状態を監視・変更したり、直接コールバックを登録したりすることを禁止する。
  * **イベント経由の更新**: 状態の変更通知はすべて `Game.Publisher.Publish` を介して配信し、ビュー側は `Game.Publisher.Subscribe` によって受け取ったイベントのペイロード情報のみを元に描画を行わなければならない。

#### ④ Prototype (プロトタイプ)
* **適用箇所**:
  * ES5における `inherits` ユーティリティによるクラス継承。
* **コーディング制約**:
  * **継承方法の統一**: 現在のコードベースが ES5 準拠のコンパイルを前提としている間は、標準の `inherits(Child, Parent)` 構造を使用する。ES6 `class` 構文の全面移行が完了するまでは、個別の ES6 クラス宣言とプロトタイプ継承の混在を避けること。

#### ⑤ Singleton (シングルトン)
* **適用箇所**:
  * シングルトンのアンチパターン（グローバルな密結合）を避けるために導入された **Service Locator** (`Game.Locator`)。
* **コーディング制約**:
  * **グローバルシングルトンの直接生成禁止**: `var MyManager = new ...` のようなグローバルシングルトン変数を新たに宣言することを禁止する。すべてのグローバルなサービスやマネージャー（`Publisher`, `MasterData`, `SceneDirector` 等）は、必ず `Game.Locator` に登録し、そこから取得する形式を徹底すること。

### 2.2 Sequencing Patterns (順序性パターン)

#### ⑥ Double Buffer (ダブルバッファ)
* **適用箇所**:
  * `Game.RenderCommandExecuter` による描画コマンドのバッチ処理。
* **コーディング制約**:
  * **即時 DOM 反映の禁止**: `Update()` フェーズ内で、直接 `document.body` や各 DOM 要素の追加・書き換えを行うことを厳禁とする。
  * **バッチ描画の徹底**: 描画の変更は `DocumentFragment` などを構築した上で描画コマンドとして `RenderCommandExecuter` にエンキューし、`OnRender` イベント時に一括して DOM に反映（フラッシュ）させなければならない。これにより画面のチラつき（フリッカー）や頻繁なリフローを防止する。

#### ⑦ Game Loop (ゲームループ)
* **適用箇所**:
  * `Engine.prototype.Loop` による固定更新/可変描画ループ（「Fix Your Timestep!」に基づく設計）。
* **コーディング制約**:
  * **独自タイマーの禁止**: ゲームオブジェクト内で個別に `setInterval` や生の `requestAnimationFrame` を用いて定期更新処理を記述してはならない。すべての更新処理は `Engine` が駆動する `Update` および `LastUpdate` ライフサイクルメソッドに集約させること。
  * **ラグ制限の遵守**: タブがバックグラウンドに回った際の累積ラグ対策（ISSUE-06）など、`Engine` に定義されたラグクランプ処理を変更してはならない。

#### ⑧ Update Method (アップデートメソッド)
* **適用箇所**:
  * `GameObject` クラスが持つ `Start`, `Update`, `LastUpdate`, `Render`, `Destroy` ライフサイクル。
* **コーディング制約**:
  * **ライフサイクルの巡回登録**: 毎フレーム更新や描画が必要なすべてのゲーム要素（エンティティ、エフェクト、UIレイヤー）は、`GameObject` を継承し、対応するメソッドをオーバーライドしなければならない。また、それらは親となる `GameObject` の `objects` 配列に登録され、木構造の巡回処理に含まれるようにすること。

### 2.3 Behavioral Patterns (行動パターン)

#### ⑨ Bytecode (バイトコード)
* **適用箇所**:
  * `template.js` による Tornado 互換テンプレートのパースと JS 関数へのコンパイル。
* **コーディング制約**:
  * **テンプレートのキャッシュ化**: 動的 HTML 生成時、生の HTML テンプレート文字列を毎フレームその場でパースしてはならない。必ず `templates.js` にて事前にコンパイルしたテンプレート関数を定義・キャッシュし、描画時にデータを流し込む設計にすること。

#### ⑩ Subclass Sandbox (サブクラスサンドボックス)
* **適用箇所**:
  * `GameObject` や `ILayer` を継承した各種カスタムレイヤー設計。
* **コーディング制約**:
  * **親クラスインターフェースの活用**: 派生したレイヤーやエンティティは、グローバル関数や無関係なモジュールを直接呼び出すのではなく、基本クラスが提供するメソッドやプロパティ、または `Game.Locator` を介して提供される共有の機能のみを利用して実装すること。これにより、具象クラスの責務を最小限にし、テストを容易にする。

#### ⑪ Type Object (タイプオブジェクト)
* **適用箇所**:
  * `PlayCard` における `CardType`（`StepCard`, `RankCard`, `DashCard`）や `Model` データによる振る舞いの分岐。
* **コーディング制約**:
  * **過度なクラス継承の回避**: カードの効果や馬の種類といった「データによってバリエーションが変化するもの」に対して、バリエーションごとに具象 JavaScript クラスを新規に作成することを避ける。マスターデータ (`MasterMeta` / `Model`) に定義されたパラメータを読み込み、共通のクラスが振る舞いを動的に決定する設計とすること。

### 2.4 Decoupling Patterns (デカップリングパターン)

#### ⑫ Component (コンポーネント)
* **適用箇所**:
  * `Race` エンティティと `GameBoard`, `Racetrack`, `OddsTable` などの構成関係、および `Lane` と `LaneRenderer` の分離。
* **コーディング制約**:
  * **神オブジェクトの禁止**: 単一の巨大なクラスに「データ・物理位置・描画・ルール判定」のすべてを詰め込んではならない。それぞれを独立したクラスやコンポーネントに分割し、コンポジション（包含）によって組み立てること。特に「ロジックコンポーネント」と「描画コンポーネント」は分離を徹底する。

#### ⑬ Event Queue (イベントキュー)
* **適用箇所**:
  * `RenderCommandExecuter` による描画指示キュー。
* **コーディング制約**:
  * **即時実行の回避**: パフォーマンスボトルネックになりやすい重い処理や、順序を保証して実行したいイベントは、即時実行せずキュー（Queue）に登録し、次の適切なフレーム更新フェーズで一括処理する設計とすること。

#### ⑭ Service Locator (サービスロケーター)
* **適用箇所**:
  * `Game.Locator` による依存解決。
* **コーディング制約**:
  * **明示的なブートストラップ登録**: すべての共通サービス・リポジトリ・ディレクターは、アプリケーション起動時の `Game.Bootstrap` 処理内において `Game.Locator.register` を用いて明示的に登録されなければならない。モジュール内でグローバル空間から暗黙的に取得してはならない。

### 2.5 Optimization Patterns (最適化パターン)

#### ⑮ Data Locality (データ局所性)
* **適用箇所**:
  * `GameObject` 内の `this.objects` 配列による連続メモリアクセス。
* **コーディング制約**:
  * **フラットな配列ループ**: 毎フレーム更新を行う対象オブジェクトは、木構造を巡回する際、ポインタの参照解決を何度も往復するような複雑なリンクリスト構造を避け、可能な限りフラットな連続配列として保持し、リニアに走査すること。

#### ⑯ Dirty Flag (ダーティフラグ)
* **適用箇所**:
  * イベント購読時のみ再描画を行うレイヤー設計。
* **コーディング制約**:
  * **無駄な再レンダリングの排除**: 状態に変化がない限り、DOM 要素の破棄と再生成を行ってはならない。データが更新されたことを示すイベント（`Events.Race.OnChanged` など）の通知を受けるか、明示的な「更新フラグ」が立った場合にのみ描画処理をトリガーすること。

#### ⑰ Object Pool (オブジェクトプール)
* **適用箇所**:
  * ログメッセージレイヤー（`LogMessageLayer`）やエフェクト生成（将来実装）。
* **コーディング制約**:
  * **GCスパイクの防止**: 頻繁に生成・破棄が繰り返される UI 要素（例：スクロールするログ行、ポップアップ、パーティクル）については、毎回 `new` や `document.createElement` を呼び出すのではなく、事前にプールしたオブジェクトを再利用（非表示・表示の切り替えおよびデータの書き換え）すること。これによりガベージコレクションによる一瞬のフレーム落ち（スパイク）を防ぐ。

#### ⑱ Spatial Partition (空間分割)
* **適用箇所**:
  * `Racetrack` のレーン設計。
* **コーディング制約**:
  * **二重ループによる全件当たり判定の禁止**: 馬と馬の近接判定や順位計算を行う際、単純な $O(n^2)$ の二重ループによる総当たりチェックを避け、レーン単位のインデックス（空間分割）やソート済み配列から特定の近接座標のみを取り出す探索アルゴリズムを採用すること。

## 3. コーディング上の具体的制約

名著のデザインパターンを適切に適用し、バニラJS環境でプロジェクトを安全に進行するため、以下の制約を厳格に課します。

### 3.1 JavaScript 基本制約
1. ** `"use strict";` の必須**:
   * すべての JavaScript ファイルの 1 行目には必ず `"use strict";` を記述すること。
2. **変数宣言規約**:
   * 現在のコンパイラ互換方針に基づき、変数宣言には `var` を使用する。ただし、今後のリファクタリング方針（ISSUE-05）に従い、移行可能な箇所は型安全性を考慮して順次移行する。
3. **JSDoc アノテーションの徹底**:
   * Google Closure Compiler による厳格な型チェックと最適化を有効にするため、すべてのコンストラクタ、プロトタイプメソッド、関数、および主要なメンバ変数に詳細な JSDoc アノテーションを記述すること。
   * 特に関数引数・戻り値の型（`@param`, `@return`）、インターフェースの実装（`@implements`）、継承関係（`@extends`）は必須とする。

### 3.2 セキュリティ制約 (XSS対策)
1. ** `innerHTML` の使用全面禁止**:
   * XSS（クロスサイトスクリプティング）脆弱性を排除するため、`Element.innerHTML` への直接代入は一切禁止する。
   * 代わりに、`document.createElement`, `document.createTextNode` などの安全な DOM 標準 API、またはテンプレートエンジンによってコンパイル・エスケープされた DOM フラグメントを使用すること。
2. ** `eval()` の使用全面禁止**:
   * セキュリティリスクと実行速度低下を防ぐため、`eval()` の使用を全面的に禁止する。文字列から動的実行を行う場合は、テンプレートエンジン内部のように隔離された環境での `new Function()` のみの使用にとどめること。

### 3.3 描画とパフォーマンスの制約
1. **フレームタイムの厳守**:
   * `Update()` や `Render()` メソッド内の処理時間は、60FPS維持のため 1 フレームあたり **16.6ms** を大幅に下回るように設計すること。
2. **DOM書き換えのバッチ化**:
   * レイアウトのリフロー（再計算）を最小限にするため、一時的な DOM 構築には `DocumentFragment` を使用し、最終的に一括で親要素にアタッチすること。
