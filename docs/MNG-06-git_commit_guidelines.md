# [MNG-06] Gitコミット規約・運用ガイドライン (Git Commit Conventions & Guidelines) - horse-racing-game-js

本ドキュメントは、「`horse-racing-game-js`」プロジェクトにおけるGitコミットの規約、コメントの記述形式、およびローカルで管理されている各種Issue（課題・不具合）ファイルとの連携ルールを定義したものです。

---

## 1. 目的と基本原則

本規約は、以下の目的を達成するために制定されています。
1. **履歴の可読性向上**: コミットログを一目で理解できるようにし、バグの混入箇所や機能追加の履歴を容易に追跡可能にします。
2. **自動化・ツール連携の準備**: [Conventional Commits](https://www.conventionalcommits.org/) 規格に準拠することで、将来的な自動チェンジログ生成やリリース自動化を容易にします。
3. **ローカルIssue管理との整合性**: Gitのコミット履歴と、`docs/issue/` ディレクトリ配下で管理される各種課題ドキュメントを直接結びつけ、進捗状況の透明性を確保します。

---

## 2. コミットメッセージのフォーマット (git commit -m 構文)

コミットメッセージは、以下の標準構造に従って記述します。単一の `-m` オプションで記述する場合も、このヘッダー（1行目）の形式を厳格に守ってください。

```bash
git commit -m "<type>(<scope>): <summary>"
```

### 2.1 構成要素の説明

1. **`<type>` (変更の種類)**: 必ず以下のいずれかの種類を指定します。
   
   | Type | 用途 | 具体例 |
   | :--- | :--- | :--- |
   | **`feat`** | 新機能の実装・追加 | `feat(bet): コイン賭けUIの実装` |
   | **`fix`** | バグ・不具合の修正 | `fix(racetrack): カード使い切り時のフリーズ修正` |
   | **`docs`** | ドキュメント類（Markdown）のみの作成・変更 | `docs(design): 設計パターンの制約を明文化` |
   | **`refactor`** | 動作変更を伴わないコードのリファクタリング | `refactor(event): イベント探索アルゴリズムの整理` |
   | **`perf`** | パフォーマンス向上を目的としたコード変更 | `perf(renderer): DOM書き換え回数の削減による高速化` |
   | **`test`** | テストコードの追加・変更・整理 | `test(template): エラー行番号検出のテスト拡充` |
   | **`chore`** | ビルドツール、パッケージ設定、補助ツールの変更 | `chore(makefile): Closure Compiler の警告オプション追加` |
   | **`balance`** | ゲームバランス調整用のパラメータの変更 | `balance(odds): 馬の初期オッズ計算式の調整` |

2. **`<scope>` (変更の適用範囲)**: 影響を受けるコンポーネントやディレクトリ名を記述します（省略可能ですが、記述を推奨します）。
   * 例: `bet`, `render`, `template`, `event`, `compiler`, `engine` など。

3. **`<summary>` (変更の簡潔な要約)**:
   * 変更内容を簡潔に記述します。
   * 日本語で記述する場合は、簡潔かつ明確な表現（例：「〜の実装」「〜の修正」）を使用してください。
   * 英語で記述する場合は、命令形かつ三人称単数現在形の `-s` をつけない（例：`add bet UI`）ようにします。

## 3. 複数行コミットメッセージの記述ルール

コマンドラインで詳細な説明を含めたい場合や、Gitエディタを使用する場合は、以下の複数行フォーマットを使用します。

```text
<type>(<scope>): <summary>

[Body (本文) - なぜこの変更を行ったのか、技術的な意思決定の背景]

[Footer (フッター) - 関連するIssue番号や参照情報]
```

* **Body (本文) の記述事項**:
  * 「何をしたか」だけでなく、「**なぜその方法で解決したか**（技術的トレードオフ、意思決定）」を記述します。
  * 既存のコードをどのように破壊・改修したかを記述します。
* **Footer (フッター) の記述事項**:
  * 関連する Issue などの外部ソース、あるいは依存関係のある仕様書へのリンクを記述します。

## 4. ローカルIssue管理との連携ルール

本プロジェクトでは、Issueトラッカーの代わりに `docs/issue/` ディレクトリ配下に `ISSUE-XX-description.md` というMarkdownファイルを配置して不具合・タスクを管理しています。そのため、コミットメッセージとこれらのファイルの連携が重要になります。

### 4.1 Issue IDのヘッダーへの埋め込み

コミットメッセージ内で、必ず関連するローカルIssue ID (`[ISSUE-XX]`) を明記してください。
特に、コミットメッセージの要約（summary）の冒頭（コロンの直後）に `[ISSUE-XX]` を配置する形式を厳格に適用します。

#### ① `-m` 一行で記述する場合の例
```bash
git commit -m "fix(racetrack): [ISSUE-01] カード使い切り時のフリーズを修正"
```

#### ② 複数行で記述する場合の例 (ヘッダー部に含める場合)
```bash
git commit -m "feat(checker): [ISSUE-03] マスターデータの値検証器 (ValueChecker) を実装

メタデータに記載された型・制約（min, regex等）に基づき起動時に自動スキャンを行う構造。
これにより不正なデータ定義によるクラッシュを早期に検知できる。"
```


### 4.2 Issue解決コミット時のルール
不具合やタスクが解決（Resolved）した際は、以下の作業を**同一のコミットに含めるか、関連するブランチのマージ前までに完了**させなければなりません。
1. **Issueファイル自体の更新**:
   * 該当する `docs/issue/ISSUE-XX-description.md` を開き、ステータスを `解決済 (Resolved)` に書き換え、対応内容と日付、関連ファイルを明記します。
2. **課題管理台帳の更新**:
   * [docs/MNG-03-problem_management.md](MNG-03-problem_management.md) の台帳レコードを更新し、ステータスを `解決済 (Resolved)` に変更します。

## 5. コミット実行前の必須チェックリスト (Best Practices)

コミットを行う前に、ターミナルで以下のチェックを必ず実行してください。

1. **テストの通過確認**:
   * コミット前に必ず `make test` を実行し、**テストが100%通過（全件PASS）すること**を確認してください。テストが失敗する状態でのコミットは厳禁です。
2. **ビルドの通過確認**:
   * `make` コマンドを実行し、コンパイルエラーや Closure Compiler の警告が発生しないことを確認してください。
3. **コミット対象ファイルの確認**:
   * `git status` でコミット対象を確認してください。ビルド結果の生成物である `main-min.js` および `main-min.css` は `.gitignore` に登録されているため通常は含まれませんが、誤ってステージングされていないか確認してください。
4. **コミットの責務分割**:
   * 「リファクタリング」と「機能追加」や、「無関係なバグ修正」を単一のコミットに混在させないでください。コミットの粒度は、1つの目的（1つのIssue）に対して1コミットとすることを推奨します。
